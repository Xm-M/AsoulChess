# 架构设计与影响面分析规范

## 规范目的

避免AI Coding的"盲目开发"风险，确保每个功能都经过充分的架构设计和影响评估。

## 强制要求

### 触发条件

**必须执行分析的情况**：
- ✅ 所有新功能开发
- ✅ 修改现有核心系统
- ✅ 性能关键路径改动
- ✅ 跨模块的功能

**豁免条件**（可不执行分析）：
- ⚠️ 简单Bug修复（< 10行代码）
- ⚠️ 配置文件调整
- ⚠️ UI文案修改
- ⚠️ 纯资源替换

### 执行时机

- **时间点**：需求评审后、开发前
- **负责人**：架构师或技术负责人
- **审查方式**：人工审查，不能AI自动完成

### 输出物

**必须输出**：
1. 架构设计与影响面分析报告（Markdown格式）
2. 包含决策审批签名区
3. 存档路径：`docs/architecture/analysis/`

## 分析内容

### 一、系统定位

**必须分析**：
1. **所属模块**：明确功能归属的模块
2. **模块关系**：
   - 上游依赖：依赖哪些模块
   - 下游影响：影响哪些模块
   - 横向关联：与哪些模块有交互

### 二、设计决策

**必须明确**：
1. **设计模式**：使用什么设计模式
2. **选择理由**：为什么选择此模式
3. **备选方案**：其他可行方案及放弃理由
4. **数据流向**：数据如何流转
5. **扩展性设计**：预留的扩展点

### 三、影响面分析

**必须评估**：
1. **改动范围**：需要修改的现有文件
2. **依赖关系**：新功能依赖哪些系统、会影响哪些系统
3. **性能影响**：CPU/内存/IO开销评估
4. **兼容性**：是否影响现有功能、是否需要版本兼容

### 四、风险识别

**必须识别**：
1. **风险清单**：潜在风险点及应对措施
2. **技术债务预警**：是否引入新的技术债务

## 分析流程

### 第一步：系统定位分析
使用 `@architecture-analyze` Skill自动分析：
- 识别所属模块
- 分析模块关系
- 生成模块依赖图

**确认点**：请用户确认模块归属是否正确

### 第二步：设计决策建议
AI提供设计建议：
- 推荐设计模式及理由
- 备选方案及优缺点对比
- 数据流向设计
- 扩展性建议

**确认点**：请用户选择设计方案

### 第三步：影响面评估
AI分析影响范围：
- 需要修改的文件列表
- 依赖关系梳理
- 性能影响评估
- 兼容性检查

**确认点**：请用户确认是否继续或调整方案

### 第四步：风险识别
AI识别潜在风险：
- 风险点列表
- 风险概率和影响
- 应对措施建议

**确认点**：请用户确认风险可接受

### 第五步：生成分析报告
AI生成完整报告：
- 输出Markdown格式报告
- 自动存档至 `docs/architecture/analysis/`
- 包含审批签名区

## 审批流程

### 审批人
1. **架构师**：审核架构合理性
2. **技术负责人**：审核影响面可控性

### 审批结果
- ✅ **通过**：可以开始开发
- ⚠️ **修改后通过**：根据意见修改后可开发
- ❌ **驳回**：方案不可行，需重新设计

### 审批标准

**架构师审查**：
- [ ] 设计模式选择合理
- [ ] 模块划分清晰
- [ ] 扩展性考虑充分
- [ ] 符合项目架构风格

**技术负责人审查**：
- [ ] 改动范围可控
- [ ] 风险识别充分
- [ ] 应对措施可行
- [ ] 测试计划完善

## 违规处理

### 未经分析直接开发
- ❌ 代码Review不予通过
- ⚠️ 需补全分析报告
- ⚠️ 纳入绩效考核（严重者）

### 分析报告不完整
- ⚠️ 需补充缺失内容
- ⚠️ 重新审批

### 审批未通过强行开发
- ❌ 代码合并被拒绝
- ⚠️ 纳入绩效考核
- ⚠️ 团队通报批评

## 使用示例

### 示例：开发背包系统

```bash
# 触发分析
@architecture-analyze
需求: 开发一个背包系统，支持物品存储、拖拽移动、堆叠合并
模块: 物品系统
```

**AI自动执行**：
1. 分析系统定位 → 确认模块归属
2. 提供设计建议 → 用户选择方案
3. 评估影响范围 → 用户确认风险
4. 生成分析报告 → 存档备查

**审批流程**：
1. 提交架构师审核
2. 提交技术负责人审核
3. 审批通过后开始开发
